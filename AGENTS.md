# 「学习之路」项目协作规则

本文件是 Codex 在本项目中执行任务时应遵守的协作规范。具体产品需求和技术设计分别参考 `PRD.md` 与 `TECH_DESIGN.md`。

## 1. 项目目标

- 保持网站简单、清楚，适合阅读和长期维护。
- 优先记录真实学习过程和可复用成果。
- 不主动扩大 MVP 范围。

## 2. 技术限制

- 继续使用 HTML、CSS 和原生 JavaScript。
- 不主动引入 React、Vue、Next.js、构建工具、数据库或后端。
- 不添加外部依赖，除非任务明确要求并经过用户确认。

## 3. 修改原则

- 开始前先阅读任务涉及的文件。
- 一次只完成一个明确的小任务。
- 只修改任务指定的文件。
- 如果必须修改其他文件，先说明原因并等待用户确认。
- 保留现有内容，不得覆盖或删除旧学习记录和复利内容。
- 新增内容必须使用真实材料，不得虚构记录、成果、数据或功能。
- 不顺便进行任务未要求的重构、美化或功能扩展。

## 4. 页面结构

- 每个 HTML 页面只能有一个 `h1`。
- 首页、每日学习和复利工程三个页面保持统一导航和页脚。
- 页面内部锚点必须唯一、目标存在且可以正常跳转。
- 新学习记录沿用 `daily.html` 中的现有结构。
- 最新记录放在 `daily-list` 列表最前面，完整正文追加到已有记录之后。
- 首页的每日学习预览只展示最新记录，并链接到对应的正确锚点。

## 5. 样式与移动端

- 保持浅色背景、深灰文字和一种低饱和强调色。
- 保持内容宽度和排版适合长时间阅读。
- 不使用复杂动画、大面积渐变或与阅读无关的视觉效果。
- 修改页面或样式后，检查窄屏显示和横向溢出。

## 6. 安全与部署

- 所有网页资源和页面链接使用相对路径。
- 不使用本机绝对路径。
- 不在代码或公开仓库中保存 API Key、密码或其他敏感信息。
- 不修改 GitHub Pages 配置，除非任务明确要求。

## 7. 验收要求

每次完成任务后说明：

- 修改了哪些文件。
- 每个文件修改了什么。
- 是否存在超出任务范围的改动。
- 相关链接和锚点是否有效。
- 是否仍符合每页一个 `h1`、内容真实和移动端基础阅读要求。

## 8. 工作节奏

1. 先分析，再修改。
2. 先完成最小可用改动，再进行验收。
3. 发现问题先列出，不直接扩大修复范围。
4. 当前任务通过验收后，再进入下一项任务。
