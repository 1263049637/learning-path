第 1 天：规划“学习之路”网站

日期：2026 年 7 月 11 日

今日学习主题：规划“学习之路”个人 AI 学习记录与复利工程网站。

今日目标

明确网站定位、页面范围、内容结构、复利工程分类和第一版 MVP 范围。

今日完成内容

  1. 确定网站名称为“学习之路”。
  2. 明确网站由学习记录、复利资料库和作品展示三部分组成。
  3. 完成首页、每日学习、周度总结、复利工程和我的作品五个主要页面规划。
  4. 完成全站导航、页脚和手机端基础要求。
  5. 设计每日学习和周度总结的固定模板。
  6. 设计提示词、Skills、工作流、方法、工具和模板六类复利内容。
  7. 完成主要页面的线框结构和点击关系。
  8. 明确第一版不使用后台、数据库、复杂动画和复杂搜索。

今日最重要的三个收获

  1. 写代码前需要先明确需求

    网站不能只从“我想做什么”开始，还需要明确目标用户、内容结构、页面关系、第一版范围和不做的功能。

  2. 学习过程和可复用成果需要分开保存

    每日学习负责记录过程，周度总结负责提炼重点，复利工程负责保存可以再次使用的成果，我的作品负责展示实践项目。

  3. 固定模板可以降低长期维护难度

    每日记录、周总结、提示词、Skill 和工作流都使用统一结构，可以减少每次整理内容时的思考成本。

今日留下的成果

一份完整的《学习之路网站规划 v2》和一份《网站线框图与交互结构 v1》。

当前需要注意的问题

网站的长期规划比较完整，但第一轮开发范围偏大。正式写代码时，需要先完成首页、每日学习和复利工程的核心浏览流程，再逐渐增加周度总结、作品详情、分类筛选和其他功能。

下一步行动

  1. 确定第一轮开发的页面范围。
  2. 准备首页、一篇每日记录和三条复利内容的真实文案。
  3. 把需求拆成小任务交给 Codex。
  4. 先完成网站结构，再逐步调整样式和内容。

完成“学习之路”第一版并部署上线

日期：2026 年 7 月 19 日

学习主题：使用 Codex 小步完成“学习之路”静态网站，并通过 GitHub Pages 部署上线。

今日目标

完成“学习之路”第一版 MVP 的开发、验收和部署，跑通从产品需求到线上网站的完整流程。

今日完成内容

  1. 将网站 PRD 和技术设计文档交给 Codex，让 Codex 先理解项目范围和开发顺序。
  2. 创建 index.html、daily.html、compound.html、styles.css、script.js 和 README.md 六个项目文件。
  3. 分别完成首页、每日学习页面和复利工程页面的真实内容结构。
  4. 为三个页面建立统一的导航、页脚、页面内部锚点和返回入口。
  5. 完成全站基础样式和手机端基础适配。
  6. 对页面范围、链接、锚点、内容真实性和 HTML 结构进行静态验收。
  7. 在浏览器中手动检查三个页面和手机端显示效果。
  8. 更新 README，整理项目说明、文件结构、当前功能和后续计划。
  9. 检查项目中的资源引用，确认网页使用相对路径，具备静态部署条件。
  10. 创建 GitHub 仓库，将六个项目文件上传并提交。
  11. 开启 GitHub Pages，将本地网站发布为可以公开访问的线上网站。

今日最重要的三个收获

  1. Codex 适合被拆成小任务指挥

    最初我对 Codex 的操作方式并不熟悉，容易把它理解成一次性生成完整项目的工具。实际开发中，我逐渐学会了限定任务范围，例如一次只创建文件结构、只修改一个 HTML 页面或只优化 CSS。

    这样不仅更容易检查结果，也能避免 Codex 扩大项目范围或修改不相关的文件。

  2. 项目完成不等于代码生成结束

    网站页面生成后，还需要继续检查页面范围、导航链接、内部锚点、内容完整性、手机端显示和部署路径。

    通过“先检查、不修改”的验收方式，我能够先发现问题，再判断是否需要修复，而不是在没有明确目标时继续增加功能。

  3. 本地网站和线上网站之间还需要发布流程

    本地文件只能在自己的电脑中打开。要让其他人访问，需要把项目上传到 GitHub 仓库，再使用 GitHub Pages 部署。

    我也理解了相对路径的重要性：网页中的 CSS、JavaScript、页面链接和锚点不能依赖自己电脑上的 D: 盘路径。

今日留下的成果

  1. “学习之路”第一版静态网站。
  2. 一个包含六个项目文件的 GitHub 仓库。
  3. 一个可以通过 GitHub Pages 访问的线上网站。
  4. Codex 项目启动提示词。
  5. Codex 小步执行 SOP。
  6. Codex 单文件修改提示词。
  7. 静态网站 MVP 验收清单。
  8. 部署前路径检查方法。

当前需要注意的问题

  1. 目前网站仍然依靠手动修改 HTML 添加学习记录和复利内容。
  2. 每次修改本地文件后，还需要把更新提交到 GitHub，线上网站才会同步变化。
  3. 当前内容数量较少，暂时不需要搜索、筛选、数据库或独立详情页。
  4. 后续需要继续熟悉 GitHub 中的提交记录、文件更新和版本管理。
  5. 网站虽然已经完成基础手机端检查，但以后每次调整样式后仍需要重新检查窄屏显示。

下一步行动

  1. 把本次建站和部署过程添加为新的学习记录。
  2. 更新首页的最新学习预览。
  3. 完成第一次网站内容更新并提交到 GitHub。
  4. 验证 GitHub Pages 是否自动更新。
  5. 整理“本地修改—提交 GitHub—线上更新”的维护流程。
  6. 在真实重复任务中继续寻找适合 Skill 化的流程。
